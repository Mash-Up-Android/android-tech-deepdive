# 1. 컴포즈의 핵심 동작 원리

### Q. 컴포즈가 데이터를 받아 실제 화면의 픽셀로 렌더링하기까지 내부적으로 거치는 3단계 과정

  > 컴포즈가 데이터를 받고 화면에 픽셀로 렌더링하는 과정은 크게 Composition, Layout, Drawing의 3단계로 이뤄진다

- **1단계: Composition (컴포지션)** - 무엇을 보여줄 것인가? (UI 트리 생성)
- **2단계: Layout (레이아웃)** - **어디에** 배치할 것인가? (Measure & Place, 크기 측정과 위치 결정)
- **3단계: Drawing (드로잉)** - **어떻게** 그릴 것인가? (Canvas, 픽셀 렌더링)

**Composition(컴포지션)** 단계

- 관찰하고 있는 State를 기반으로 어떤 `@Composable` 함수들을 실행할지 결정하여 **UI 트리**를 구축한다
- 화면에 **'무엇을 보여줄지'** 결정하는 단계

**Layout(레이아웃)** 단계

- 만들어진 UI 트리를 순회하면서 각 UI 요소의 크기를 측정하고, 화면의 어느 좌표에 배치할지 결정한다
- 화면의 **'어디에 위치시킬지'** 정하는 과정

**Drawing(드로잉)** 단계

- 앞서 계산된 크기와 위치 정보를 바탕으로 기기의 캔버스에 픽셀 단위로 UI를 실제로 그린다
- 최종적으로 **'어떻게 화면에 나타낼지'** 렌더링하는 단계


### 스마트 리컴포지션

- 상태가 변경되었을 때 화면 전체를 다시 그리는 것이 아니라, **실제로 데이터가 변경된 컴포저블(UI 노드)만 선택적으로 다시 실행**하고 변경되지 않은 부분은 렌더링을 건너뛰는 컴포즈 최적화 기법


컴포즈는 상태가 변경되었을 때 무조건 1단계부터 다시 시작하는 것이 아니다

크기나 위치만 바뀌었다면?

1단계를 건너뛰고 2단계나 3단계부터만 다시 실행하도록 설계되어 있어 렌더링 성능이 매우 뛰어나다

컴포즈가 특정 컴포저블의 렌더링을 건너뛸지 말지 결정하는 핵심 기준
- 매개변수의 안정성(Stability)
- 구조적 동등성(Equality)

리컴포지션이 트리거되면, 컴포즈 런타임은 해당 컴포저블에 들어오는 이전 입력값과 새로운 입력값을 `==` 연산자로 비교한다

이때 입력값이 String이나 Int 같은 원시 타입이거나 데이터가 변하지 않음이 보장되는 불변(Immutable/Stable) 객체이고, 비교 결과 값이 이전과 완벽히 동일하다면 해당 컴포저블의 재실행을 안전하게 **건너뛴다**

만약 매개변수로 일반적인 `List`나 `var`가 포함된 클래스와 같이

내부 데이터가 언제든 변할 수 있는 불안정(Unstable)한 객체가 들어오면?

컴포즈는 변화를 놓치는 버그를 막기 위해 값이 같아 보여도 매번 리컴포지션을 강제로 수행한다

따라서 스마트 리컴포지션이 정상적으로 동작하게 하려면 상태를 최대한 불변(Immutable) 객체로 관리하는 것이 중요


### **Q. 리컴포지션(Recomposition)이란?**

Recomposition

- 관찰 중인 State가 변경되었을 때 최신 데이터를 반영하기 위해 컴포저블 함수를 다시 실행하여 UI 트리를 갱신하는 과정

앱이 처음 실행되어 초기 컴포지션이 일어난 이후,

컴포저블 내부에서 읽고 있는 `State` 객체의 값이 업데이트되는 시점에 바로 리컴포지션이 발생한다

여기서 특징은 컴포즈가 스마트 리컴포지션을 수행한다는 점

상태가 변했다고 해서 화면 전체를 처음부터 끝까지 다 다시 그리는 것이 아니라, 변경된 상태를 참조하는 특정 컴포저블 함수들만 찾아내어 재실행한다

반대로, 파라미터가 바뀌지 않은 나머지 컴포저블들은 실행을 **건너뛰어** 불필요한 리소스 낭비를 막고 렌더링 성능을 최적화한다


### Q. 상태 호이스팅(State Hoisting)

상태 호이스팅

- 컴포저블 내부에 있는 **상태를 호출하는 부모 쪽으로 끌어올리는 설계 패턴**

내부에서 `remember`로 상태를 직접 들고 있는 대신,

**현재 상태를 나타내는 값 + 상태를 변경해 달라고 요청하는 콜백 함수**(Event)를 매개변수로 받도록 구조를 변경하는 것

이 패턴을 사용해야 하는 이유

- 재사용성과 **단방향 데이터 흐름(UDF)의 구축**

상태를 위로 끌어올리면, 해당 컴포저블은 스스로 상태를 가지지 않는 순수한 **Stateless 컴포저블**이 된다

덕분에 컴포넌트가 특정 상태 관리 로직에 종속되지 않는다

또한 여러 화면에서 자유롭게 재사용할 수 있고, UI 테스트를 하기도 훨씬 수월하다

결론적으로 상태 호이스팅을 통해

**상태는 위에서 아래로 흐르고, 이벤트는 아래에서 위로 올라가는**

깔끔한 UDF 구조를 완성할 수 있다


# 2. 성능 최적화 및 상태 저장

### Q. 불필요한 리컴포지션을 방지하기 위한 최적화 기법

불필요한 리컴포지션을 방지하여 렌더링 성능을 최적화하기 위해 실무에서는 크게 3~4가지 기법을 주로 활용한다

**데이터 모델의 안정성(Stability) 보장**

일반적인 `List`나 `var`가 포함된 데이터 클래스는 Unstable하여 항상 리컴포지션을 유발한다

불변성 보장하는 `@Immutable` 어노테이션을 달거나, 컬렉션의 경우 `ImmutableList`를 사용하여 컴포즈가 안전하게 렌더링을 건너뛸 수 있도록 만든다

**`derivedStateOf`의 활용**

스크롤 오프셋처럼 상태가 1픽셀 단위로 변할 때마다 화면을 다시 그리면 성능이 크게 저하된다

이때 `derivedStateOf`를 사용하면 '스크롤이 100px을 넘었는가?'처럼 **내가 진짜 관심 있는 Boolean 조건이 변할 때만** 리컴포지션이 일어나도록 필터링할 수 있다

**`LazyColumn` 등 리스트에서 `key`를 지정**

리스트 중간에 아이템이 추가되거나 삭제될 때 `key`가 없으면 아래의 모든 아이템이 다시 그려진다

고유 식별자(key)를 지정해주면 변경된 아이템만 재배치되어 성능이 크게 향상된다


### **Q. `remember`와 `rememberSaveable`의 차이점은 무엇인가요?**


  `remember`와 `rememberSaveable`은 둘 다 리컴포지션이 발생할 때 상태 값이 초기화되는 것을 막고 이전 값을 기억하기 위해 사용한다

  가장 큰 차이점은 상태의 생존 범위에 있다

  `remember`는 컴포지션 내의 메모리에만 상태를 저장

    - 일반적인 리컴포지션 상황에서는 값을 잘 유지하지만, 기기를 가로로 회전하거나 다크모드를 전환하는 등 Configuration Change가 발생하여 액티비티가 재시작되면? 저장되어 있던 상태가 모두 날아감.

  `rememberSaveable`은 내부적으로 안드로이드의 **`Bundle`** 메커니즘을 활용하여 상태를 저장

    - 화면이 회전하거나, 시스템 리소스 부족으로 앱 프로세스가 잠시 죽었다가 살아나는 상황에서도 상태를 안전하게 저장하고 **복원**할 수 있다

  단순한 UI 애니메이션 상태 같은 휘발성 데이터는 `remember`로 처리

  사용자가 입력 중인 검색어나 텍스트 필드의 값, 스크롤 위치처럼 화면이 돌아가도 반드시 유지되어야 하는 중요한 데이터는 `rememberSaveable`을 사용

# 3. 부수 효과 (Side Effect) 및 생명주기

### **Q. 컴포즈에서 Side Effect란 무엇이며, 대표적인 Side Effect API(`LaunchedEffect`, `DisposableEffect` 등)의 차이점은 무엇인가요?**


컴포즈에서 Side Effect란,

화면을 그리는 컴포저블 함수 외부의 앱 상태를 변경하는 모든 행위

컴포저블은 입력이 같으면 결과가 같은 '순수 함수'여야 한다

하지만 실무에서는 스낵바를 띄우거나, 데이터를 불러오고 리스너를 등록하는 등의 외부 작업이 필수적

리컴포지션이 일어날 때마다 이런 작업들이 중복 실행되는 것을 막기 위해 컴포즈엔 다음과 같은 대표적인 Side Effect API가 있다

`LaunchedEffect`

컴포저블 내부에서 비동기 작업(코루틴)을 안전하게 실행

컴포저블이 화면(Composition)에 진입할 때 코루틴을 실행하고

화면에서 사라지거나 `key` 값이 변경되면 자동으로 코루틴을 취소한다

Ex) 스낵바 띄우기, 뷰모델에 초기 데이터 로딩 요청

`DisposableEffect`

작업 이후 Cleanup이 반드시 필요할 때 사용

`LaunchedEffect`와 달리 코루틴 스코프를 제공하지 않으며, 컴포저블이 화면에서 사라질 때 **`onDispose`** 블록을 호출하여 자원을 해제한다

Ex) Lifecycle 옵저버 등록, 메모리 누수를 막기 위한 이벤트 리스너 해제



# 4. 아키텍처 및 패러다임 비교

### **Q. 기존 XML 기반의 명령형 UI와 Compose의 선언형 UI의 가장 큰 차이점**

  가장 큰 차이는

  State를 대하는 패러다임

  기존 XML 기반의 명령형 UI에서는 데이터가 변경될 때마다 뷰를 직접 참조한다

  `textView.text = "..."`, `button.visibility = View.GONE`처럼

  개발자가 화면을 어떻게 바꿀지 일일이 명령하고 제어해야 한다

  → 화면이 복잡해질수록 UI와 실제 데이터의 상태가 어긋나는 동기화 버그가 발생하기 쉽다

  Compose의 선언형 UI는 **UI = f(State)** 라는 공식처럼 동작한다

  데이터가 주어지면 그 상태에 맞는 화면이 무엇인지 선언하기만 하면 된다

  → 개발자가 뷰를 직접 수정할 필요 없이, 상태만 업데이트해주면 프레임워크가 알아서 변경된 상태에 맞춰 화면을 다시 그려준다 → 리컴포지션

  개발자가 화면의 상태 동기화라는 복잡한 작업에서 벗어나, 데이터 흐름 자체에만 집중할 수 있게 만들어준다는 것이 선언형 UI의 큰 장점
