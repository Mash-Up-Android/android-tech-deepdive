# MVI 기본 개념 및 동작 원리

### **Q. MVI 아키텍처의 3가지 요소(Model, View, Intent)**
MVI 아키텍처

- 단방향 데이터 흐름(UDF)을 기반으로 복잡한 UI 상태를 안전하게 관리하는 패턴

3가지 핵심 요소는 다음과 같다

**Model**

- 단순한 데이터가 아니라 화면의 현재 ‘**상태'** 그 자체
- 로딩, 성공, 에러 등의 상태를 **불변 객체**로 관리하기 때문에 상태가 꼬이는 것을 막아준다

**View**

- Model을 **관찰**하고 있다가 새로운 상태가 들어오면 화면을 업데이트 한다
- 특히 상태를 기반으로 UI를 다시 그리는 **Jetpack Compose** 환경에서 큰 장점

**Intent**

- 버튼 클릭과 같은 사용자의 액션, 이벤트
- 안드로이드 컴포넌트의 Intent 아님

안드로이드의 Intent

- 안드로이드 애플리케이션의 4대 컴포넌트(Activity, Service, Broadcast Receiver 등) 간에 작업을 요청하고, 데이터를 주고받기 위해 사용하는 메시지 전달 객체

사용자가 Intent를 발생시키면,
비즈니스 로직을 거쳐 새로운 Model이 만들어지고
View가 이를 렌더링하는 **단방향 흐름**

덕분에, 데이터 변화가 잦고 복잡한 화면도 버그 추적 및 디버깅이 훨씬 수월하다


### **Q. 단방향 데이터 흐름(UDF, Unidirectional Data Flow)**

단방향 데이터 흐름(UDF)

- 데이터가 오직 한 방향으로만 파이프라인을 타고 흐르도록 설계된 패턴
- 핵심 원리 : State는 ViewModel에서 View(아래)로 흐르고, Intent는 View에서 ViewModel(위)로 흐른다

이 구조가 가져다주는 가장 큰 장점

- 예측 가능성
- 디버깅의 용이성

View와 ViewModel이 서로 데이터를 수정하게 되면,
화면이 복잡해졌을 때 데이터가 꼬이거나 동기화 문제가 발생하기 쉽다

하지만 UDF 환경에서는 상태를 쥐고 있는 단일 진실 공급원(SSOT)이 존재하고,
오직 사용자의 이벤트만이 새로운 상태를 만들어낼 수 있다

View는 그저 전달받은 최종 상태를
화면에 그리기만 하면 되기 때문에 UI가 항상 일관되게 유지되며
버그가 발생하더라도 이벤트가 발생한 지점만 역추적하면 되므로
디버깅과 테스트가 훨씬 수월해진다

> SSOT

- 어떤 데이터나 상태는 오직 단 하나의 위치에서만 생성, 관리, 수정되어야 한다는 원칙
- 다른 곳에서 어떤 데이터가 필요하다면
  복사본을 만들어 따로 수정하는 것이 아니라, 무조건 그 단일 출처를 참조(Observe)해야 한다

안드로이드에서는 **보통 `ViewModel`이 SSOT의 역할을 담당**

- View는 화면에 데이터를 띄우고 사용자의 클릭을 받을 뿐,
  '현재 체크박스가 체크되어 있는지', '로딩 중인지' 같은 **상태를 스스로 저장하고 판단해서는 안됨**
- ViewModel에게 "사용자가 체크박스 누름" 이벤트만 보내면 SSOT인 ViewModel이 `isChecked = true`로 상태를 업데이트하고,
  View는 바뀐 상태를 Observe하고 있다가 화면에 표시하는 것

# 아키텍처 비교

### **Q. 기존에 많이 사용하던 MVVM 패턴과 MVI 패턴의 가장 큰 차이점**

> MVVM과 MVI의 가장 큰 차이점 == UI 상태(State)를 관리하는 방식
>

**MVVM 패턴**

- 화면 로딩 상태, 데이터 로드 성공, 에러 등
  상태를 각각 분리된 여러 개의 `LiveData`나 `StateFlow`로 관리
- 간단한 화면에서는 빠르고 좋다!
  하지만, 화면이 복잡해질수록 상태들이 서로 얽히면서 의도치 않은 UI 버그가 발생하기 쉽고
  데이터 흐름을 추적하기 어려워진다
-

**MVI 패턴**

- 화면의 모든 상태를 묶어 단일 상태 객체로 관리
- 사용자의 액션인 Intent가 발생하면,
  기존 상태를 직접 수정하는 것이 아니라 새로운 상태 객체를 복사해 만들어
  View로 내려주는 **엄격한 단방향 흐름**을 강제한다



MVVM은 유연하지만 상태가 분산될 위험이 있는 반면,

MVI는 상태를 하나의 불변 객체로 통제하여 **디버깅과 상태 예측이 훨씬 수월하다**

하지만 초기 보일러플레이트 코드가 많아진다는 단점이 존재…

### **Q. MVVM에서도 단방향 데이터 흐름(UDF)을 구현할 수 있는데, 굳이 MVI를 도입해야 하는 이유**

- 답안

  MVVM 패턴에서도 단일 `StateFlow`를 활용하면 UDF를 충분히 구현할 수 있다

  그럼에도 굳이 MVI를 도입한다면 가장 큰 이유?


  1. 구조적 강제성과 **Intent의 명시적 정의**

    - MVVM은 UDF를 '권장'할 뿐 강제하지는 않는다
    - 실무에서 개발 일정에 쫓기다 보면 ViewModel 안에 여러 개의 상태 변수를 추가하게 되고,
      결국 다시 상태가 파편화되는 경우가 많아질 수 있다
    - 반면, MVI는 아키텍처 자체가 단일 상태와 단방향 흐름을 물리적으로 강제한다
      따라서, 앱이 복잡해져도 구조가 무너지지 않는다


  2. 이벤트 처리 방식의 차이

  - MVVM : 버튼 클릭이나 새로고침을 ViewModel의 개별 함수(`onClick()`, `onRefresh()`)로 처리
  - MVI : 사용자의 액션을 **Intent라는 하나의 객체로 묶어 단일 통로로 전달 →** 이벤트를 객체화하면 어떤 순서로 액션이 발생해서 현재의 상태가 되었는지 로그를 남기고 디버깅하기가 훨씬 직관적

  여러 명의 개발자가 협업하거나 상태 변화가 매우 복잡한 화면에서는,
  개발자의 재량에 의존하는 MVVM보다
  엄격하고 명확한 규칙을 제공하는 MVI가 유지보수와 안정성 면에서 훨씬 유리하다고 생각

# 실무 적용 및 트러블슈팅

### **Q. MVI 패턴에서 토스트 메시지, 스낵바 노출, 화면 이동과 같은 Side Effect 처리**

이를 처리하는 핵심은
지속적으로 화면에 보여야 하는 상태(State)와,
한 번 발생하고 소모되어야 하는 **일회성 이벤트**를 엄격하게 **분리**하는 것

만약 토스트 메시지 노출 여부를 일반적인 State 객체 안에 포함시킨다면?

기기를 회전하거나, Jetpack Compose에서 Recomposition이 일어날 때
마지막 상태가 다시 불리면서 토스트가 중복해서 뜨는 버그가 발생한다

따라서 ViewModel에서 분리하여 설계한다

- UI 상태: 값을 유지하는 `StateFlow`로 관리
- 일회성 이벤트: 한 번 전달되면 소비되어 사라지는 `Channel`이나 `SharedFlow` 활용

### **Q. MVI 패턴을 도입했을 때 느낄 수 있는 단점이나 한계점**

1. 보일러플레이트 코드의 증가
    - 아주 단순한 화면을 개발한다고 해도
      UI State, Intent, 그리고 Side Effect를 처리하기 위한 클래스를 일일이 정의해야 해서
      초기 세팅 시간이 오래 걸린다
    - 불변 객체를 사용하기 때문에
      상태가 조금만 변해도 `copy()`를 통해 **새로운 객체를 계속 생성해야 하는
      미세한 메모리 오버헤드**도 발생할 수 있다
2. 높은 러닝 커브
    - 코루틴의 `StateFlow`나 `Channel` 같은 비동기 스트림 등
      팀원들의 높은 이해도가 요구된다

결론적으로 모든 화면에 강박적으로 MVI를 고집하기보다는,
상태 변화가 복잡한 핵심 화면에 우선 적용하는 식의 **유연한 도입**이 필요

이러한 한계를 완화하기 위해…

**BaseViewModel**

- 설계상태와 이벤트를 처리하는 공통 파이프라인을 **추상화하여** 반복되는 코드를 줄인다
- 안드로이드 스튜디오의 템플릿 기능을 활용하는 등
