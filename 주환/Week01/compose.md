## 📌 Compose Compiler의 역할은 무엇인가요?
Composable 함수를 Runtime이 실행 중에 활용할 수 있는 형태로 구조적으로 변환해주는 역할을 합니다.

Composable 함수에 숨겨진 파라미터(Composer, changed 등)를 자동으로 삽입해준다거나..

안정성 분석을 통해 recomposition이 필요한지를 판단한다거나...

스킵 가능한 함수인지 판단하여 불필요한 재실행을 방지합니다.

## 📌 remember랑 mutableStateOf에 대해서 알려주세요.

remember는 값을 Composition에 저장하고, mutableStateOf는 값 변경 시 리컴포지션을 트리거합니다.

좀 더 풀어서 설명하자면...

compose를 화가라고 한다면, 이 화가는 그림을 그리기 전에 메모장(slot table)에 모든 값을 적어둡니다.

어떤 요소가 어디에 그려질것이고 어쩌구 저쩌구...

그리고 나중에 값이 변경되면 그림을 처음부터 다시 그리는 것이 아니라 해당 부분만 다시 그리는거죠.

즉, remember는 메모장에 이 값을 적어줘! 라는 뜻이고

mutableStateOf는 이 값이 변경되면 나에게 알려줘! 라는 뜻입니다.

## 📌 remember와 mutableStateOf는 왜 함께 사용되어야 하나요?

mutableStateOf만 사용하면 상태값이 바뀔 때마다 recomposition을 트리거하게 되는데,

이렇게 되면 메모리에 값이 저장되어 있지 않기 때문에 초기화되고, 기대했던 동작과 다른 결과가 나오게 됩니다.

따라서 상태 또한 메모리에 저장을 해야 의미가 있기에 같이 사용해야 합니다.

## 📌 Recompose의 매커니즘을 설명해주세요.

Recomposer가 코루틴 기반의 루프를 돌며 무효화된 Scope들을 모아 프레임 단위로 일괄처리 합니다.

매 vsync마다 InvalidationQueue를 확인해서 변경된 RecomposeScope만 선택적으로 재실행합니다.

### 🤔 vsync가 뭘까?

vsync는 디스플레이 하드웨어가 화면을 그릴 준비가 되었다고 보내는 신호임

60fps면 1초에 60번 보내는거고... 그때마다 안드로이드는 콜백을 받는데

Recomposer도 거기에 등록해두는거임 "화면 그릴 준비가 되면 나도 알려줘!" 라고...

### 🤔 Recomposer는 코루틴 기반의 루프를 무한정으로 돌고있는걸까?

맞음! 실제로도 while(true)로 되어있음

코루틴의 suspend 함수를 사용하기 때문에

할 일이 없으면 CPU 점유 없이 대기하여 문제가 되진 않음

### 🤔 InvalidationQueue는 뭘까?

무효화된 Scope들을 모아두는 곳!

모아두었다가 화면 그릴 준비가 되면 한 번에 일괄 처리하는 것임

같은 프레임 안에서 여러 상태가 동시에 바뀔 수 있기 때문에 이렇게 처리하는 것임

## 📌 Compose가 화면을 그리는 순서는 어떻게 되나요?

Composition, Layout, Drawing 단계로 이루어집니다.

Composition에서는 UI 트리를 구축하고

Layout에서는 컴포넌트를 정렬하고

Drawing에서는 모든 것을 시각적으로 렌더링합니다.

## 📌 선언형 UI란 무엇인가요?

UI가 어떻게 그려지는지(How)가 아니라, 어떤 상태에서 어떻게 보여야 하는지(What)를 선언하는 방식입니다.

## 📌 선언형 UI가 명령형 UI와 비교했을 때 어떤 장점이 있나요?

코드가 간결해집니다. XML 레이아웃 없이 Kotlin 코드 하나로 UI를 완성할 수 있습니다.

또, 컴포저블 함수 단위로 쪼개서 조합하기 쉬워 재사용성이 좋습니다.

Preview를 지원한다는것도 장점입니다. 한 화면에 많은 상태 케이스가 담겨 있는 경우 XML에서는 이를 다 표현할수가 없었는데 Compose에서는 가능합니다.

그리고 상태 관리 버그를 줄일 수 있다는 것도 장점입니다.

XML을 사용하는데 뷰의 상태를 변경하는 포인트가 여러곳이라면.. 그 코드들이 흝어져있다면 어떤 원인으로 상태가 변경되는지 파악하기가 쉽지 않습니다.

하지만 compose를 사용하면 그런 상황을 막을 수 있습니다.

### 🤔 XML이 정말 명령형 UI가 맞는건가...? XML도 뷰의 상태를 묘사하고 있지 않나?

XML만 놓고 본다면 XMl도 선언형 UI임!

어떤 상태에서 어떻게 보여야 하는지 정의해뒀잖아?

근데 Compose랑 달리 XML은 Koltin이나 Java 코드로 뷰 객체의 상태를 변경해야함

이것이 가장 큰 차이점이고 XML은 수동적인 동기화가 요구된다고 볼 수 있음

즉, 안드로이드에서 XML이 명령형 UI라고 하는건 XML 자체를 말하는게 아니라 XML을 제어하는 View 시스템을 포함하여 명령형 UI이다 라고 말하는것임

## 📌 불필요한 Recompose를 막기 위해선 어떻게 해야하나요?

State를 필요한 Composable에만 가깝게 위치시켜 영향 범위를 줄입니다.

LazyColumn에 키를 설정한다거나

Stable과 Immutable 어노테이션을 활용해 스킵 최적화를 적용하는 등의 방법이 있습니다.

다만...

Layout Inspector로 실제 병목을 먼저 확인하고 적용하는 게 중요하다고 생각합니다.

섣부른 최적화는 코드 복잡도만 높인다거나 사이드 이펙트를 만들어낼 수 있기 때문입니다. (특히 어노테이션 !!)
